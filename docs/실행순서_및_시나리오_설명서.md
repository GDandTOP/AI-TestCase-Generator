# TestPlanner 실행 순서 및 시나리오 설명서

> **전제 조건**: 깃허브(또는 로컬 저장소) 불러오기가 완료되었고, 비교 브랜치(또는 커밋/최근 N개) 설정이 끝난 상태입니다.

이 문서에서는 **「분석 시작」 버튼을 누른 이후**의 실행 순서를, 데이터 예시와 시나리오와 함께 정리합니다.

---

## 1. 전체 실행 순서 (한눈에)

```
[설정 완료] 저장소 경로 + 비교 대상(base/head 또는 커밋, 최근 N개) + AI 모델 선택
        │
        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │ 1단계: Diff 추출 (POST /api/git/diff)                            │
  │        → 서버가 Git에서 변경 내용(diff) 가져옴                    │
  └─────────────────────────────────────────────────────────────────┘
        │
        ▼  diffResult 저장
  ┌─────────────────────────────────────────────────────────────────┐
  │ 2단계: 영향도 분석 (POST /api/analysis/impact)                   │
  │        → Claude AI가 diff를 보고 위험도·영향 영역·권고사항 반환   │
  └─────────────────────────────────────────────────────────────────┘
        │
        ▼  impactAnalysis 저장 → Step 2 화면(영향도 결과) 표시
  ┌─────────────────────────────────────────────────────────────────┐
  │ [사용자] "테스트케이스 생성" 클릭                                 │
  └─────────────────────────────────────────────────────────────────┘
        │
        ▼
  ┌─────────────────────────────────────────────────────────────────┐
  │ 3단계: 테스트케이스 생성 (POST /api/testcase/generate, SSE)       │
  │        → Claude가 TC 마크다운을 스트리밍으로 생성                 │
  └─────────────────────────────────────────────────────────────────┘
        │
        ▼  headerContent + tcContent 저장 → Step 3 화면(TC 뷰어) 표시
  ┌─────────────────────────────────────────────────────────────────┐
  │ [선택] 저장 → POST /api/testcase/save → output/*.md               │
  │ [선택] 다운로드 → GET /api/testcase/download/:filename            │
  │ [선택] 처음으로 → reset()                                        │
  └─────────────────────────────────────────────────────────────────┘
```

---

## 2. 1단계: Diff 추출 (POST /api/git/diff)

**언제**: 사용자가 Step 1에서 **「분석 시작」** 을 클릭한 직후.

**목적**: 선택한 비교 대상(base↔head, 커밋 범위, 최근 N개 커밋)에 대한 **Git 변경 내용(diff)** 을 서버에서 가져와 화면·다음 API에 쓸 수 있게 합니다.

### 2.1 요청 데이터 예시

비교 방식에 따라 `compareType`과 `base`/`head` 또는 `recentCount`가 정해집니다.

| 비교 방식 | compareType | 요청 body 예시 |
|-----------|-------------|----------------|
| **브랜치 비교** | `branch` | `repoPath`, `compareType: "branch"`, `base: "main"`, `head: "feature/login"` |
| **커밋 2개 비교** | `commit` | `repoPath`, `compareType: "commit"`, `base: "a1841ae"`, `head: "8238e26"` |
| **최근 N개 커밋** | `recent` | `repoPath`, `compareType: "recent"`, `recentCount: 5` |

**브랜치 비교 시 요청 예시 (JSON):**

```json
{
  "repoPath": "/tmp/cloned-repo-abc123",
  "compareType": "branch",
  "base": "main",
  "head": "feature/login"
}
```

**최근 5개 커밋 비교 시 요청 예시:**

```json
{
  "repoPath": "/tmp/cloned-repo-abc123",
  "compareType": "recent",
  "recentCount": 5
}
```

### 2.2 응답 데이터 예시 (GitDiffResult)

서버는 아래와 같은 형태로 diff 결과를 반환합니다.

```json
{
  "success": true,
  "data": {
    "files": [
      { "path": "src/App.tsx", "status": "modified", "insertions": 12, "deletions": 3 },
      { "path": "src/api/client.ts", "status": "added", "insertions": 45, "deletions": 0 }
    ],
    "rawDiff": "diff --git a/src/App.tsx b/src/App.tsx\nindex 123..456 100644\n--- a/src/App.tsx\n+++ b/src/App.tsx\n@@ -10,3 +10,5 @@\n ...",
    "stats": {
      "filesChanged": 2,
      "insertions": 57,
      "deletions": 3
    }
  }
}
```

- **files**: 변경된 파일 목록(경로, 상태, 추가/삭제 라인 수).
- **rawDiff**: `git diff` 원문. 크기 제한(예: 50,000자) 초과 시 잘릴 수 있음.
- **stats**: 전체 변경 통계.

이 `data`가 스토어의 **diffResult** 로 저장되고, 다음 단계(영향도 분석)의 입력으로 사용됩니다.

---

## 3. 2단계: 영향도 분석 (POST /api/analysis/impact)

**언제**: 1단계 diff 추출이 성공한 직후, 자동으로 호출됩니다.

**목적**: Claude AI가 **diff만 보고** 전체 위험도, 영향 받는 영역, 테스트 시 권고사항을 JSON으로 반환합니다.

### 3.1 요청 데이터 예시

클라이언트는 **1단계에서 받은 diff** 와 **선택한 AI 모델** 을 보냅니다.

```json
{
  "diff": {
    "files": [
      { "path": "src/html/login.html", "status": "modified", "insertions": 36, "deletions": 10 }
    ],
    "rawDiff": "diff --git a/src/html/login.html ...",
    "stats": { "filesChanged": 1, "insertions": 36, "deletions": 10 }
  },
  "model": "claude-haiku-4-5-20251001"
}
```

서버 내부에서는 이 diff를 **프롬프트 문자열** 로 만든 뒤, Claude API에 `messages: [{ role: 'user', content: prompt }]` 형태로 전달합니다.

### 3.2 응답 데이터 예시 (ImpactAnalysis)

Claude가 다음 JSON 형식으로 답합니다. 이 전체가 **impactAnalysis** 로 저장됩니다.

```json
{
  "success": true,
  "data": {
    "overallRisk": "medium",
    "summary": "로그인 화면의 스타일과 레이아웃이 변경되었습니다. 버튼 색상·폰트가 바뀌어 시각적 회귀 테스트가 필요합니다.",
    "affectedAreas": [
      {
        "name": "로그인 화면",
        "risk": "medium",
        "description": "로그인 페이지의 버튼·입력창 스타일이 변경됨. 동일 URL에서 화면이 다르게 보일 수 있음.",
        "files": ["src/html/login.html"]
      }
    ],
    "recommendations": [
      "로그인 후 메인 화면 진입이 정상인지 확인",
      "비밀번호 오류 시 메시지가 올바르게 표시되는지 확인"
    ]
  }
}
```

- **overallRisk**: `low` | `medium` | `high` | `critical`
- **affectedAreas**: 영향 받는 기능/화면별 이름, 위험도, 설명, 관련 파일.
- **recommendations**: 수동 테스트 시 꼭 할 일 목록.

이후 **Step 2 화면**으로 전환되며, 위 내용이 요약·목록 형태로 표시됩니다.

## 4. 3단계: 테스트케이스 생성 (POST /api/testcase/generate, SSE)

**언제**: 사용자가 Step 2에서 **「테스트케이스 생성」** 버튼을 클릭한 뒤.

**목적**: diff + 영향도 분석 결과를 바탕으로 Claude가 **QA용 테스트케이스 마크다운**을 **실시간 스트리밍(SSE)** 으로 생성합니다.

### 4.1 요청 데이터 예시

diff, 영향도 분석 결과, 프로젝트명, 비교 요약, 모델을 함께 보냅니다.

```json
{
  "diff": { "files": [...], "rawDiff": "...", "stats": {...} },
  "analysis": {
    "overallRisk": "medium",
    "summary": "...",
    "affectedAreas": [...],
    "recommendations": [...]
  },
  "projectName": "시간외근무 기록 시스템",
  "compareSummary": "main...feature/login",
  "model": "claude-haiku-4-5-20251001"
}
```

- **compareSummary**: 화면에 보이는 비교 설명(예: `main...feature/login`, `최근 5개 커밋`).

### 4.2 응답 형식 (SSE 스트리밍)

HTTP 응답은 **text/event-stream** 이며, 줄 단위로 다음처럼 옵니다.

| 이벤트 type | 의미 | 예시 |
|-------------|------|------|
| `header` | 보고서 상단(제목·비교 정보 등) | `data: {"type":"header","text":"# 테스트케이스\n\n- 비교: main...feature/login\n"}\n\n` |
| `delta` | TC 본문 조각(연속 전송) | `data: {"type":"delta","text":"## 1. 로그인\n\n| 단계 | ..."}\n\n` |
| `done` | 스트림 종료 + 토큰 사용량 | `data: {"type":"done","usage":{...}}\n\n` |
| `error` | 오류 시 | `data: {"type":"error","text":"..."}\n\n` |

클라이언트는 `header` → `headerContent`, `delta` → **tcContent에 누적** 하여 실시간으로 화면에 렌더링합니다. `done` 수신 후 **Step 3 화면**으로 전환됩니다.

## 5. 시나리오별 정리

### 시나리오 A: 브랜치 비교 (main vs feature/login)

1. **설정**: compareType=`branch`, base=`main`, head=`feature/login`.
2. **실행 순서**  
   - `POST /api/git/diff` → body: `repoPath`, `compareType: "branch"`, `base: "main"`, `head: "feature/login"`.  
   - diff 수신 → `POST /api/analysis/impact` (diff + model).  
   - 영향도 수신 → Step 2 표시.  
   - 사용자가 「테스트케이스 생성」 클릭 → `POST /api/testcase/generate` (diff, analysis, projectName, compareSummary: `"main...feature/login"`, model).  
   - SSE로 TC 수신 → Step 3 표시.

### 시나리오 B: 커밋 2개 비교 (a1841ae..8238e26)

1. **설정**: compareType=`commit`, base=`a1841ae`, head=`8238e26`.
2. **실행 순서**  
   - `POST /api/git/diff` → body: `repoPath`, `compareType: "commit"`, `base: "a1841ae"`, `head: "8238e26"`.  
   - 이후는 시나리오 A와 동일.  
   - compareSummary는 예: `"a1841ae..8238e26"` 형태로 표시.

### 시나리오 C: 최근 5개 커밋

1. **설정**: compareType=`recent`, recentCount=`5`.
2. **실행 순서**  
   - `POST /api/git/diff` → body: `repoPath`, `compareType: "recent"`, `recentCount: 5`.  
   - 이후 동일.  
   - compareSummary는 예: `"최근 5개 커밋"`.

## 6. Step 3 이후 선택 동작 (데이터 예시)

| 사용자 동작 | API / 동작 | 데이터 예시 |
|-------------|------------|-------------|
| **MD 파일 저장** | `POST /api/testcase/save` | body: `content`(전체 마크다운), `diff`, `analysis`, `projectName`, `compareSummary` → 서버가 `output/테스트케이스-2026-03-09-프로젝트명.md` 형태로 저장 후 파일명 반환. |
| **다운로드** | `GET /api/testcase/download/:filename` | 이미 저장된 파일명으로 다운로드. |
| **처음으로** | (API 없음) | `store.reset()` → Step 1로 돌아가며 상태 초기화. |

---

## 7. 요약 표: API 호출 순서와 입출력

| 순서 | API | 입력(요청) | 출력(응답/저장) |
|------|-----|------------|-----------------|
| 1 | POST /api/git/diff | repoPath, compareType, base/head 또는 recentCount | GitDiffResult → **diffResult** |
| 2 | POST /api/analysis/impact | diff, model | ImpactAnalysis → **impactAnalysis**, Step 2 표시 |
| 3 | POST /api/testcase/generate | diff, analysis, projectName, compareSummary, model | SSE(header/delta/done) → **headerContent**, **tcContent**, Step 3 표시 |
| (선택) | POST /api/testcase/save | content, diff, analysis, projectName, compareSummary | 저장된 파일명 → **savedFilename** |
| (선택) | GET /api/testcase/download/:filename | - | MD 파일 다운로드 |

실제 코드 흐름은 `docs/PROJECT_STRUCTURE_AND_FLOW.md` 및 `client/src/components/GitConfig/GitConfig.tsx`(분석 시작), `Analysis.tsx`(테스트케이스 생성)를 참고하면 됩니다.
