### E2E-001: 브랜치 비교 → TC 생성 → 다운로드 (정상 전체 흐름)

- **우선순위**: P1
- **카테고리**: Happy Path
- **사전 조건**:
  - 서버/클라이언트 정상 기동 상태
  - 테스트용 Git 저장소 준비 (브랜치 2개 이상, 실제 코드 변경 이력 존재)
  - `ANTHROPIC_API_KEY` 유효

**테스트 시나리오**: 사용자가 Git 저장소를 등록하고, 브랜치 간 차이를 분석하여 TC를 생성한 뒤 MD 파일로 다운로드하는 전체 흐름이 정상 동작하는지 검증한다.

**실행 단계**:
1. 브라우저에서 `http://localhost:5173` 접속
2. Step 1 화면에서 유효한 Git 저장소 경로 입력 (예: `/Users/username/my-project`)
3. **[저장소 확인]** 버튼 클릭
4. 브랜치 목록이 드롭다운에 로딩되는지 확인
5. 비교 방식 **[브랜치 비교]** 선택
6. Base 브랜치: `main`, Head 브랜치: `feature/xxx` 선택
7. 프로젝트 이름 입력 (예: `my-project`)
8. **[분석 시작]** 버튼 클릭
9. 로딩 메시지 "diff 추출 중..." → "Claude AI로 영향도 분석 중..." 순서로 표시 확인
10. Step 2 화면으로 자동 전환 확인
11. 전체 위험도, 변경 통계, 영향 파일 목록, 영향 받는 기능 영역 표시 확인
12. **[테스트케이스 생성 (AI 스트리밍)]** 버튼 클릭
13. Step 3 화면으로 자동 전환 확인
14. "AI가 테스트케이스를 생성하고 있습니다..." 스피너 표시 확인
15. TC 내용이 실시간으로 마크다운 렌더링되며 스트리밍되는지 확인
16. 스트리밍 완료 후 스피너 사라지고 버튼 3개 표시 확인: [클립보드 복사], [MD 파일 저장], (저장 후) [다운로드]
17. **[MD 파일 저장]** 클릭 → "저장 완료: my-project_2026-02-27T..." 메시지 확인
18. **[다운로드]** 클릭 → `.md` 파일 다운로드 시작 확인
19. 다운로드된 파일 열어 보고서 헤더(문서 정보, 변경사항 요약, 영향도 분석) + TC 내용 + 체크리스트 포함 확인

**예상 결과**:
- 전체 흐름이 오류 없이 완료됨
- 생성된 MD 파일에 TC-001, TC-002 등 번호 매겨진 테스트케이스 최소 5개 이상 포함
- P1/P2/P3 우선순위가 구분되어 있음
- `output/` 디렉토리에 파일 실제 저장 확인

### E2E-002: 커밋 비교 방식으로 TC 생성

- **우선순위**: P1
- **카테고리**: Happy Path
- **사전 조건**: 테스트용 저장소에 커밋 이력 2개 이상 존재

**테스트 시나리오**: 브랜치 대신 특정 커밋 해시 두 개를 기준으로 diff를 추출하여 TC를 생성한다.

**실행 단계**:
1. Step 1에서 저장소 경로 입력 후 **[저장소 확인]** 클릭
2. 비교 방식 **[커밋 비교]** 선택
3. Base 커밋 드롭다운에서 과거 커밋 선택 (예: `abc1234 - Fix login bug`)
4. Head 커밋 드롭다운에서 최신 커밋 선택 (예: `def5678 - Add user profile`)
5. **[분석 시작]** 클릭
6. Step 2 영향도 분석 결과 확인
7. TC 생성까지 E2E-001의 12~19번 단계 동일하게 진행

**예상 결과**:
- 선택한 두 커밋 사이의 변경사항만 분석됨
- 보고서의 "분석 기준" 항목에 `abc1234..def5678` 형식으로 표시됨

### E2E-003: 최근 N개 커밋 비교 방식으로 TC 생성

**예상 결과**:
- 보고서 "분석 기준" 항목에 `최근 3개 커밋` 형식으로 표시됨

### E2E-004: 잘못된 저장소 경로 입력

- **우선순위**: P1
- **카테고리**: 오류 처리

**테스트 시나리오**: 존재하지 않는 경로를 입력했을 때 사용자에게 명확한 오류 메시지를 표시하는지 검증한다.

**실행 단계**:
1. Step 1에서 존재하지 않는 경로 입력 (예: `/Users/nobody/fake-project`)
2. **[저장소 확인]** 클릭
3. 화면 하단 오류 메시지 영역 확인

**예상 결과**:
- 빨간색 오류 박스에 `"유효하지 않은 Git 저장소입니다"` 메시지 표시
- 브랜치 드롭다운이 나타나지 않음
- **[분석 시작]** 버튼이 노출되지 않음
- 페이지 이동 없음 (Step 1 유지)

### E2E-005: Git 저장소가 아닌 일반 디렉토리 입력

- **우선순위**: P1
- **카테고리**: 오류 처리

**테스트 시나리오**: 실제로 존재하지만 `.git` 폴더가 없는 디렉토리를 입력했을 때 오류 처리를 확인한다.

**실행 단계**:
1. Git으로 초기화되지 않은 디렉토리 경로 입력 (예: `/Users/username/Documents`)
2. **[저장소 확인]** 클릭

**예상 결과**:
- `"유효하지 않은 Git 저장소입니다"` 오류 메시지 표시
- 이후 단계로 진행 불가

### E2E-006: diff가 없는 동일 브랜치 비교

- **우선순위**: P2
- **카테고리**: 경계값

**테스트 시나리오**: Base와 Head에 동일한 브랜치를 선택했을 때 빈 diff 상황을 올바르게 처리하는지 확인한다.

**실행 단계**:
1. 저장소 확인 후 비교 방식 **[브랜치 비교]** 선택
2. Base 브랜치: `main`, Head 브랜치: `main` (동일하게) 선택
3. **[분석 시작]** 클릭
4. Step 2 영향도 분석 결과 확인

**예상 결과**:
- 변경된 파일 수: **0개** 표시
- 전체 위험도: **낮음** 표시
- 영향도 요약: `"변경사항이 없거나 분석할 diff가 없습니다."` 표시
- TC 생성 시 빈 diff에 대한 안내 메시지 포함

### E2E-007: 매우 큰 diff (MAX_DIFF_SIZE 초과)

- **우선순위**: P2
- **카테고리**: 경계값

**테스트 시나리오**: 50,000자를 초과하는 대용량 diff가 발생했을 때 서버가 자동으로 잘라내고 사용자에게 안내하는지 확인한다.

**사전 조건**: 대규모 변경사항이 있는 저장소 (예: 수백 개 파일 변경)

**실행 단계**:
1. 대용량 변경사항이 있는 저장소 경로 입력 및 확인
2. 브랜치 비교로 대규모 diff 추출
3. Step 2 영향도 분석 결과 확인

**예상 결과**:
- diff가 50,000자에서 잘리고 `"... (diff가 너무 커서 잘렸습니다)"` 메시지 포함
- 서버가 타임아웃 없이 정상 응답
- TC 생성은 잘린 diff 기준으로 진행됨

### E2E-008: 저장 후 MD 파일 다운로드

- **우선순위**: P1
- **카테고리**: 파일 저장

**테스트 시나리오**: TC 생성 완료 후 MD 파일을 서버에 저장하고 다운로드했을 때 파일 내용이 정상인지 검증한다.

**실행 단계**:
1. E2E-001 1~16번 단계 완료 (TC 생성 완료 상태)
2. **[MD 파일 저장]** 버튼 클릭
3. `"저장 완료: [파일명]"` 녹색 메시지 확인
4. **[다운로드]** 버튼 클릭
5. 다운로드된 파일을 텍스트 에디터로 열기
6. `output/` 디렉토리에서 동일 파일 존재 확인 (`ls -la output/`)

**예상 결과**:
- 파일명 형식: `[프로젝트명]_YYYY-MM-DDTHH-MM-SS.md`
- 파일 내용 구조:
  ```
  # 테스트케이스 보고서
  ## 문서 정보 (표)
  ## 1. 변경사항 요약
  ## 2. 영향도 분석
  ## 3. 테스트케이스 (TC-001 ~ ...)
  ## 테스트 실행 체크리스트
  ```
- 파일 인코딩: UTF-8
- `output/` 디렉토리에 실제 파일 생성됨

### E2E-011: ANTHROPIC_API_KEY 미설정 상태

- **우선순위**: P1
- **카테고리**: 오류 처리
- **사전 조건**: `server/.env`에서 `ANTHROPIC_API_KEY` 제거 또는 빈 값으로 설정

**테스트 시나리오**: API 키가 없을 때 서버가 시작 단계에서 오류를 발생시켜 잘못된 상태로 운영되지 않는지 확인한다.

**실행 단계**:
1. `server/.env`에서 `ANTHROPIC_API_KEY` 값을 제거
2. `npm run dev:server` 실행

**예상 결과**:
- 서버 프로세스가 시작 즉시 종료됨
- 콘솔에 `"환경변수 설정 오류: { ANTHROPIC_API_KEY: [...] }"` 오류 메시지 출력
- 서버가 비정상 상태로 기동되지 않음 (포트 3000 미점유)

### E2E-013: 뒤로 가기 후 재분석

- **우선순위**: P2
- **카테고리**: 네비게이션

**테스트 시나리오**: Step 2 또는 Step 3에서 이전 단계로 돌아간 뒤 다른 브랜치를 선택해 재분석이 가능한지 확인한다.

**실행 단계**:
1. Step 2 화면에서 **[← 뒤로]** 클릭
2. Step 1 화면으로 돌아온 것 확인 (이전 입력값 유지 확인)
3. Head 브랜치를 다른 브랜치로 변경
4. **[분석 시작]** 클릭
5. Step 2에서 변경된 브랜치 기준으로 결과가 업데이트되는지 확인

**예상 결과**:
- 뒤로 가기 후 이전에 입력한 저장소 경로, 비교 방식, 브랜치 선택값이 유지됨
- 재분석 시 이전 TC 내용이 초기화됨
- 새로운 브랜치 기준으로 분석 결과 갱신됨

### E2E-014: 프로젝트 이름 입력 후 TC 저장 파일명 확인

- **우선순위**: P2
- **카테고리**: 파일 저장

**테스트 시나리오**: 프로젝트 이름 입력 여부에 따라 저장되는 MD 파일명이 달라지는지 확인한다.

**실행 단계 (케이스 A - 프로젝트 이름 있음)**:
1. 프로젝트 이름에 `my-app` 입력 후 TC 생성 완료
2. **[MD 파일 저장]** 클릭
3. 표시되는 파일명 확인

**실행 단계 (케이스 B - 프로젝트 이름 없음)**:
1. 프로젝트 이름을 비워두고 TC 생성 완료
2. **[MD 파일 저장]** 클릭
3. 표시되는 파일명 확인

**예상 결과**:
- 케이스 A: `my-app_2026-02-27T...md` 형식
- 케이스 B: 저장소 폴더명 또는 `tc_2026-02-27T...md` 형식
- 특수문자(`/`, `:`, `*` 등)가 파일명에 포함되지 않음

## API 레벨 검증 (curl)

서버 동작을 직접 검증할 때 사용한다.

```bash
# 헬스체크
curl http://localhost:3000/health

# 저장소 유효성 검사
curl -X POST http://localhost:3000/api/git/validate \
  -H "Content-Type: application/json" \
  -d '{"repoPath": "/path/to/your/repo"}'

# 브랜치 목록 조회
curl -X POST http://localhost:3000/api/git/branches \
  -H "Content-Type: application/json" \
  -d '{"repoPath": "/path/to/your/repo"}'

# diff 추출 (브랜치 비교)
curl -X POST http://localhost:3000/api/git/diff \
  -H "Content-Type: application/json" \
  -d '{
    "repoPath": "/path/to/your/repo",
    "compareType": "branch",
    "base": "main",
    "head": "feature/my-branch"
  }'

# diff 추출 (최근 1개 커밋)
curl -X POST http://localhost:3000/api/git/diff \
  -H "Content-Type: application/json" \
  -d '{
    "repoPath": "/path/to/your/repo",
    "compareType": "recent",
    "count": 1
  }'

# TC 파일 목록 조회
curl http://localhost:3000/api/testcase/list
```
